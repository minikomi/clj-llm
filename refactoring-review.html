Refactoring clj-llm

After 19 bug-fix tasks, two passes to clean up accumulated complexity: structural decomplection, then a readability sweep across all 11 source files.

74 tests, 237 assertions, 0 failures. 9 commits, 16 files touched. 1,452 lines of source.

1. Pure state machine for consume-events

The go-loop carried five positional recur arguments with state updates and channel I/O interleaved in every case branch. Adding a sixth accumulator meant touching every recur. State transitions couldn't be tested without async channels.

Before
(go-loop [chunks []
          tool-calls []
          tc-index {}
          usage-acc {}
          finish-reason nil]
  (case (:type event)
    :content
    (do (>! text-chunks-chan (:content event))
        (recur (conj chunks (:content event))
               tool-calls tc-index
               usage-acc finish-reason))
    :usage
    (recur chunks tool-calls tc-index
           (merge usage-acc ...)
           finish-reason)
    ;; every branch repeats all 5 args
After
;; Pure function, no channels
(defn- next-state [state event]
  (case (:type event)
    :content  (update state :chunks conj ...)
    :usage    (update state :usage merge ...)
    :done     (assoc state :done? true)
    state))

;; Go-loop is just plumbing
(go-loop [state init-state]
  (if-let [event (<! source-chan)]
    (let [state' (next-state state event)]
      (>! events-chan event)
      (if (:done? state')
        (finalize! state')
        (recur state')))
    (finalize! state)))

next-state is pure and testable with plain maps. New accumulators = add a key + a case branch. finalize! takes one arg, not three.

2. Uniform data->internal-events contract

Backend event converters returned nil, a single map, or a vector of maps. The caller had to (if (sequential? result) result [result]) every time.

Before
;; Returns map OR vector
(cond
  content    {:type :content ...}
  tool-calls (if (= 1 (count events))
               (first events)
               (vec events)))

;; Caller unwraps
(let [evts (if (sequential? result)
             result [result])] ...)
After
;; Always returns seq or nil
(cond
  content    [{:type :content ...}]
  tool-calls (not-empty
               (keep convert-one tool-calls))
  :else nil)

;; Caller: no unwrapping
(if-let [evts (seq (parse-sse-data ...))]
  ...)

Both backends follow the same contract. One return shape to handle.

3. Single code path in run-agent

The agent loop had two nearly-identical branches (tool calls vs. reasoning), each with its own max-steps check, history construction, and recur. Merged into one let that branches only on values, then one check and one recur. Later extracted the step-building into build-agent-step:

(let [{:keys [history-msgs step]} (build-agent-step name->fn parsed-calls text)
      next-history (into history history-msgs)
      next-steps   (if step (conj steps step) steps)]
  (if (>= (inc n) max-steps)
    {:text text :history next-history :steps next-steps :truncated true}
    (recur next-history next-steps (inc n))))

5. Consistency

  • unwrap! helper — the check-if-exception-and-throw pattern appeared 5 times with slight variations. Now one function.
  • Parameterized validate-opts — request and run-agent both validated keys but used different code. Now one function with an optional key set arg.
  • map->Response — replaced 6-field positional ->Response. Positional construction silently breaks when fields shift.
  • Single status-errors table — errors.clj had two independent case expressions on HTTP status codes. Now one data table drives both.

6. Split malli->json-schema

At depth 0 it returned a tool definition {:type "function" :function {...}}. At depth > 0 it returned a JSON Schema {:type "object" ...}. The name said JSON Schema but half the time it returned a tool wrapper.

Now malli->json-schema always returns plain JSON Schema. malli->tool-definition wraps it for tool calling. Backends call the one they need.

7. Decouple structured output from event loop

consume-events knew about structured-promise and the output schema — it delivered errors to the promise and spawned a parsing future. Now consume-events only handles text, tool-calls, usage, and channel fan-out. The structured-output future lives in request.

8. Rename :schema → :output-schema

The :schema option key shadowed the schema namespace alias and was ambiguous (Malli schema? JSON Schema? structured output option?). Renamed across the protocol, both backends, core, tests, scripts, and README. Internal uses in error data and Malli metadata lookups unchanged.
