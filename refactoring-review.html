Why this refactor?

After 19 bug-fix tasks, the code worked correctly but had accumulated structural complexity: a go-loop with 5 positional recur args, polymorphic event return types, duplicated agent branches, cryptic names, and a schema function that was two functions in a trenchcoat. Two passes made it a joy to read.

Phase 1

Structural Decomplection

Three interconnected changes in a single commit (9a5fe75).

Pure State Machine for consume-events

The go-loop in consume-events is the heart of the library. It had grown to carry five positional recur arguments, with state updates and channel I/O interleaved in every case branch.

✘ Before — tangled state + I/O
(go-loop [chunks []
          tool-calls []
          tc-index {}
          usage-acc {}
          finish-reason nil]
  (case (:type event)
    :content
    (do (>! text-chunks-chan (:content event))
        (recur (conj chunks (:content event))
               tool-calls tc-index
               usage-acc finish-reason))
    :usage
    (recur chunks tool-calls tc-index
           (merge usage-acc ...)
           finish-reason)
    ;; every branch repeats all 5 args
✔ After — pure transitions + thin loop
;; Pure function — no channels, no promises
(defn- next-state [state event]
  (case (:type event)
    :content     (update state :chunks conj ...)
    :tool-call   (-> state (update ...) (assoc-in ...))
    :usage       (update state :usage merge ...)
    :done        (assoc state :done? true)
    state))

;; Go-loop is just plumbing
(go-loop [state init-state]
  (if-let [event (<! source-chan)]
    (let [state' (next-state state event)]
      (>! events-chan event)
      (when (= :content (:type event))
        (>! text-chunks-chan (:content event)))
      (if (:done? state')
        (finalize! state')
        (recur state')))
    (finalize! state)))
Key wins:
  • next-state is pure — testable with plain maps, no async setup
  • Adding accumulators = add key to init-state + a case branch. No recur args to update.
  • Side effects are visually separated from state logic
  • finalize! takes one arg (state), not three

Uniform data->internal-events Always Returns Seq

Backend event converters returned nil, a single map, or a vector of maps, forcing conditional unwrapping downstream.

✘ Before — polymorphic return
;; Returns map OR vector
(cond
  content    {:type :content ...}       ;; map
  tool-calls (if (= 1 (count events))
               (first events)           ;; map
               (vec events))            ;; vector!

;; Caller must unwrap
(let [evts (if (sequential? result)
             result [result])] ...)
✔ After — always seq or nil
;; Always seq
(cond
  content     [{:type :content ...}]
  tool-calls  (not-empty
                (keep convert-one tool-calls))
  finish      [{:type :finish ...}]
  usage       [(into {:type :usage} usage)]
  :else nil)

;; Caller: no unwrapping
(if-let [evts (seq (parse-sse-data ...))]
  ...)

Dedup Single Code Path in run-agent

The agent loop had two nearly-identical branches: one for tool calls, one for reasoning. Each had its own max-steps check, history construction, and recur. Merged into one let with one check and one recur.

Consistency Same Pattern, Same Code

unwrap! helper (5 call sites)

The check-if-exception-and-throw pattern appeared with slight variations at every promise deref. Now one function.

(defn- unwrap! [v]
  (if (instance? Exception v) (throw v) v))

Parameterized validate-opts

request and run-agent both validated option keys but used different code paths with different error shapes. Now one function with an optional key set argument.

(defn- validate-opts
  ([opts] (validate-opts opts known-keys))
  ([opts valid-keys]
   (let [unknown (remove valid-keys (keys opts))]
     (when (seq unknown)
       (throw (errors/error ...))))))

Single status-errors table

errors.clj had two independent case expressions mapping HTTP status codes. Now one data table drives both status->error-type and parse-http-error.

Clarity “Why” Comments at 10 Sites

Added comments where intent was non-obvious:

  • (dropping-buffer 1024) — slow consumers must not block the go-loop that delivers promises
  • (catch ExceptionInfo e (throw e)) — re-throw our errors; only catch JSON parse failures below
  • Unknown event type → state — forward-compat with new provider events
  • "required" → {:type "any"} in Anthropic — different vocabulary, not a bug
  • (:throw false) in net.cljc — errors handled via callback

Also removed dead ::type :stream-error and fixed a description "" default that existed only to be discarded.

Split malli->json-schema Dual Personality

At depth=0 it returned a tool definition {:type "function" :function {...}}. At depth>0 it returned a JSON Schema {:type "object" ...}. The function name promised JSON Schema but delivered tool wrappers half the time.

✘ Before — one function, two identities
(defn malli->json-schema [schema depth]
  ;; At depth 0: returns tool wrapper
  ;; At depth > 0: returns JSON Schema
  (if (zero? depth)
    {:type "function"
     :function {:name name
                :description desc
                :parameters base-map}}
    base-map))
✔ After — two functions, clear contracts
(defn malli->json-schema [schema]
  ;; Always returns plain JSON Schema
  {:type "object" :properties ... :required ...})

(defn malli->tool-definition [schema]
  ;; Wraps for tool calling
  {:type "function"
   :function {:name name
              :description desc
              :parameters (malli->json-schema schema)}})

Structure Decouple Structured Output from Event Loop

consume-events knew about structured-promise and schema — it delivered errors to the structured promise on :error events and spawned a future for parsing. This coupled the event consumer to a feature it shouldn't know about.

After: consume-events only knows about text, tool-calls, usage, and channel fan-out. The structured-output future moved to request, where it wraps the event loop.

Extract build-agent-step Helper

The densest block in run-agent — 7 bindings, conditional message construction, history assembly — became a named function. The loop body now reads as:

;; The entire inner loop body
(let [{:keys [history-msgs step]} (build-agent-step name->fn parsed-calls text)
      next-history (into history history-msgs)
      next-steps   (if step (conj steps step) steps)]
  (if (>= (inc n) max-steps)
    {:text text :history next-history :steps next-steps :truncated true}
    (recur next-history next-steps (inc n))))

API Rename :schema → :output-schema

The :schema option key shadowed the schema namespace alias and was ambiguous — Malli schema? JSON Schema? Structured output option? Renamed across the entire surface:

  • Protocol parameter: schema → output-schema
  • Both backends: build-body, data->internal-events, request-stream
  • Core: option key, destructuring, docstrings
  • Tests, scripts, sandbox, README

Internal uses of :schema in error data and Malli metadata lookups are unchanged — they refer to actual schemas, not the structured output option.
