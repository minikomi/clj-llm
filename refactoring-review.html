Why this refactor?

After 19 bug-fix tasks, three areas had accumulated structural complexity: a go-loop with 5 positional recur args, a polymorphic event return type (nil | map | vector), and duplicated branches in the agent loop. Each individually was fine; together they made the code harder to reason about and easier to break.

Pure 1. State Machine for consume-events

The go-loop in consume-events is the heart of the library — it reads SSE events from a source channel, accumulates text/tool-calls/usage, and delivers promises. It had grown to carry five positional recur arguments, with state updates and channel I/O interleaved in every case branch.

Problems: Adding a 6th accumulator means touching every recur call. Can't test state transitions without async channels. Side effects and state logic are interleaved.
✘ Before — tangled state + I/O
(go-loop [chunks []
          tool-calls []
          tc-index {}
          usage-acc {}
          finish-reason nil]
  (if-let [event (<! source-chan)]
    (do
      (>! events-chan event)
      (case (:type event)
        :content
        (do (>! text-chunks-chan (:content event))
            (recur (conj chunks (:content event))
                   tool-calls tc-index
                   usage-acc finish-reason))
        :usage
        (recur chunks tool-calls tc-index
               (merge usage-acc ...)
               finish-reason)
        :done
        (finalize! (apply str chunks)
                   (not-empty tool-calls)
                   (cond-> usage-acc ...))
        ;; every branch repeats all 5 args
✔ After — pure transitions + thin loop
;; Pure function — no channels, no promises
(defn- next-state [state event]
  (case (:type event)
    :content     (update state :chunks conj ...)
    :tool-call   (-> state (update ...) (assoc-in ...))
    :usage       (update state :usage merge ...)
    :finish      (assoc state :finish-reason ...)
    :error       (assoc state :done? true :error ...)
    :done        (assoc state :done? true)
    state))

;; Go-loop is just plumbing
(go-loop [state init-state]
  (if-let [event (<! source-chan)]
    (let [state' (next-state state event)]
      (>! events-chan event)
      (when (= :content (:type event))
        (>! text-chunks-chan (:content event)))
      (if (:done? state')
        (finalize! state')
        (recur state')))
    (finalize! state)))
What this buys:
  • next-state is a pure function — testable with plain maps, no async setup
  • Adding new accumulators = add a key to init-state + a case branch. No recur args to update.
  • Side effects are visually separated from state logic
  • finalize! takes one arg (state), not three

Derivation functions extract final values cleanly:

(defn- state->text [{:keys [chunks error]}]
  (or error (apply str chunks)))

(defn- state->usage [{:keys [usage finish-reason]} provider-opts req-start]
  (let [u (cond-> usage finish-reason (assoc :finish-reason finish-reason))]
    (when (seq u)
      (assoc u :type :usage
               :clj-llm/provider-opts provider-opts
               :clj-llm/req-start req-start
               :clj-llm/req-end (System/currentTimeMillis) ...))))

Uniform 2. event->internal Always Returns Seq

Backend event converters (data->internal-event in OpenAI and Anthropic) had a polymorphic return: nil, a single map, or a vector of maps. This forced backend_helpers to conditionally unwrap every result.

Problems: Caller must handle three return shapes. (if (sequential? ...) ...) is a code smell. Nested loop inside a go block is fragile with core.async macro limitations.
✘ Before — polymorphic return
;; OpenAI: returns map OR vector
(defn- data->internal-event [data schema tools]
  (cond
    content  {:type :content ...}   ;; map
    tool-calls
      (let [events (keep convert-one tool-calls)]
        (when (seq events)
          (if (= 1 (count events))
            (first events)   ;; map
            (vec events))))  ;; vector!
    ...))

;; backend_helpers: must unwrap
(if-let [result (event->internal ...)]
  (let [evts (if (sequential? result)
               result
               [result])]
    ...))
✔ After — always seq or nil
;; OpenAI: always seq
(defn- data->internal-events [data schema tools]
  (cond
    content     [{:type :content ...}]
    tool-calls  (not-empty
                  (keep convert-one tool-calls))
    finish      [{:type :finish ...}]
    usage       [(into {:type :usage} usage)]
    :else nil))

;; backend_helpers: no unwrapping
(if-let [evts (seq (event->internal ...))]
  (let [done? (loop [es evts]
                ...)]
    (when-not done? (recur)))
  (recur))
What this buys:
  • One return shape to handle — no conditional wrapping at the call site
  • The done? flag replaces (some #(= :done ...) evts) — computed during the loop, not after
  • Both backends (openai.clj, anthropic.clj) follow the same contract

Dedup 3. Single Code Path in run-agent

The agent loop had two nearly-identical branches: one for "has tool calls" and one for "no tool calls (model reasoning)". Each had its own max-steps check, history construction, and recur.

Problems: max-steps check duplicated (2×). History construction duplicated with slightly different shapes. Adding a new field to the return map means editing both branches.
✘ Before — two parallel branches
(if (empty? (or tc []))
  ;; No tool calls
  (if (>= (inc n) max-steps)
    {:text text :history ... :truncated true}
    (recur (conj history {:role :assistant
                          :content text})
           steps (inc n)))
  ;; Has tool calls
  (let [msg (tool-calls->assistant-message tc text)]
    (if (>= (inc n) max-steps)       ;; dup!
      {:text text :history ... :truncated true}
      (let [results (mapv ... tc)
            ...]
        (recur new-history
               (conj steps ...)
               (inc n))))))
✔ After — one path, branch on values
(let [has-tc?  (seq tc)
      results  (when has-tc? (mapv ... tc))
      msg      (if has-tc?
                 (tool-calls->assistant-message tc text)
                 {:role :assistant :content text})
      msgs     (if has-tc?
                 (into [msg] (mapv ... results))
                 [msg])
      next-history (into history msgs)
      next-steps   (if has-tc?
                     (conj steps {:tool-calls ...
                                  :tool-results ...})
                     steps)]
  (if (>= (inc n) max-steps)  ;; once!
    {:text text :history next-history
     :steps next-steps :truncated true}
    (recur next-history next-steps (inc n))))
What this buys:
  • One max-steps check, one recur, one return-map shape
  • New return fields go in one place
  • has-tc? is a value, not a control-flow fork
